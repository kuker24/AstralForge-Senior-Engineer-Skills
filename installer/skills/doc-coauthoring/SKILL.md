# Doc Co-Authoring

Use this skill when the user wants help producing a substantial written artifact: design docs, proposals, architecture notes, project plans, RFCs, incident writeups, implementation specs, onboarding docs, or reader-facing technical guidance. The value of this skill is not only writing text; it helps gather context, choose structure, refine language, and validate whether the final document works for its audience.

Do not use this skill for short chat replies, code-only tasks, or document formats that require a specialized file handler such as `.docx`, `.pptx`, `.xlsx`, or PDF layout work. Route those file-specific tasks to the matching skill.

## Workflow

1. Identify the reader, decision owner, business context, and desired outcome.
2. Ask for constraints: deadline, length, tone, approval process, source material, and what must not be included.
3. Propose an outline before drafting long sections.
4. Draft in small reviewable chunks with headings, bullets, examples, and explicit assumptions.
5. Run a reader test: can a new reader identify the purpose, decisions, risks, next steps, and owner?
6. Revise for clarity, evidence, and actionability.

## Output Expectations

Return the document path or the drafted section, a short summary of what changed, unresolved questions, and suggested next review steps. Prefer concise, structured writing over dense prose.

## Quality Notes

Separate facts from recommendations. Mark assumptions clearly. Avoid inventing metrics, commitments, or citations. When editing existing docs, preserve the author’s intent unless the requested task is a rewrite.
